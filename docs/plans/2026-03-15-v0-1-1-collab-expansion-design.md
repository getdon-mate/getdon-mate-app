# Getdon Mate v0.1.1 Collaboration Expansion Design

## Goal

모임 운영에 필요한 후속 UX를 상세 화면 안에서 완결되게 제공한다. 잔액 가리기 경험을 더 자연스럽게 만들고, 미납 리마인드/송금 요청, 통계, 캘린더, 게시판을 백엔드 없이도 릴리즈 수준으로 동작하게 만든다.

## Scope

- 잔액 가리기 UI를 스켈레톤/블러 계열 카드로 정교화
- `길게 눌러 잠시 표시` 개선안은 Linear 후속 태스크로 등록
- 미납 멤버 대상 리마인드/송금 요청 액션 추가
- 모임통장 상세 탭에 통계, 캘린더, 게시판 추가
- 거래 필터를 접힘 기본값 + 필터 아이콘 확장형 UI로 변경
- 웹 e2e와 unit 테스트를 확장해 회귀를 막음

## Product Direction

상세 화면은 기능이 많아져도 복잡해 보이지 않게 유지한다. 신규 기능은 전부 같은 카드 시스템과 타이포 계층 안에 넣고, 시각 효과는 과장된 장식 대신 흐림, 스켈레톤, 대비 조절로 처리한다. 게시판과 캘린더도 업무 도구에 가까운 차분한 밀도로 맞춘다.

## UX Decisions

### 1. Balance Privacy

- 잔액 가리기 상태에서는 기존 숫자 폭을 유지하는 스켈레톤 바를 사용한다.
- Hero 잔액 영역은 반투명 블러처럼 보이는 소프트 오버레이 카드로 감춘다.
- `꾹 눌러 잠시 표시`는 백로그로 분리하고, 현재 릴리즈는 안정적인 토글 UX에 집중한다.

### 2. Reminder Actions

- 미납자는 `회비`와 `멤버` 탭 어디서든 다시 찾을 수 있어야 한다.
- 액션 문구는 독촉보다 안내에 가깝게 쓴다.
- 예: `납부 안내 보내기`, `송금 요청 보내기`
- 실제 외부 전송 대신 인앱 알림/요청 로그를 생성하고 toast로 즉시 피드백한다.

### 3. Statistics

- 별도 차트 라이브러리 없이 카드형 막대 추이와 도넛 대체 시각화로 구성한다.
- 최근 3개월 수입/지출 흐름과 카테고리 분포를 우선 제공한다.
- 상세 숫자는 차트 아래 요약 카드에서 다시 읽을 수 있게 한다.

### 4. Calendar

- 월 그리드와 일정 리스트를 함께 보여준다.
- 일정 종류는 회비 마감일, 일회성 회비 마감일, 거래 발생일, 게시판 공지일로 통일한다.
- 날짜를 눌렀을 때 같은 화면 안에서 해당 일정만 확인할 수 있게 한다.

### 5. Board

- 공지/일반 글 구분을 두고, 공지는 항상 상단에 고정한다.
- 댓글은 인라인으로만 지원해 흐름을 단순화한다.
- 백엔드 전까지는 로컬 상태 기반이며, 작성 즉시 상세 화면 전체에서 반영된다.

### 6. Transaction Filters

- 거래 탭 기본 상태에서는 요약과 목록에 집중한다.
- 우상단 필터 아이콘을 눌렀을 때만 검색/유형/정렬/카테고리 필터가 펼쳐진다.
- 열린 상태는 현재 화면 세션 동안만 유지한다.

## Architecture

- `GroupAccount`에 리마인드, 게시글, 댓글, 캘린더 이벤트를 담을 최소 필드를 추가한다.
- 통계/캘린더 데이터는 기존 거래/회비 레코드에서 selector로 파생한다.
- 게시판과 리마인드 액션은 `AppProvider`에 로컬 mutation API를 추가한다.
- 상세 탭 바 타입을 확장하고, 각 탭을 독립 컴포넌트로 추가해 기존 dashboard/transactions 패턴을 재사용한다.

## Error Handling

- reminder/board mutation 실패 시 toast 대신 alert를 사용해 즉시 원인을 드러낸다.
- 공유/리마인드 액션은 낙관적 UI 없이 성공 후 상태를 갱신한다.
- 빈 통계/빈 일정/빈 게시판은 설명보다 다음 행동을 제안하는 빈 상태 카드로 처리한다.

## Testing

- 단위 테스트: 잔액 가리기 렌더링, 필터 접힘/펼침, 리마인드 액션, 통계/캘린더/게시판 탭 노출
- e2e: 신규 탭 진입, 거래 필터 열기, 게시글/댓글 작성, 미납 알림 액션
- 최종 검증: `pnpm test --runInBand`, `pnpm run ci:web`, `pnpm test:e2e`

## Release Constraints

- 백엔드가 없으므로 API 연동 대신 로컬 상태 일관성에 집중한다.
- 네이티브 런타임 크래시 위험이 있는 prop 조합은 피한다.
- 상세 탭 수가 늘어나므로 작은 화면에서도 스크롤 가능한 탭 바 또는 2열형이 아니라 단일 행 유지가 가능한 설계를 사용한다.
