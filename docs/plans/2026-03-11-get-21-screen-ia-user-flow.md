# GET-21 화면 IA / 유저 플로우 정의

## 현재 앱 기준 화면 트리

```text
App
└── login
    ├── 로그인 모드
    └── 회원가입 모드

└── account-list
    ├── 사용자 헤더
    ├── 모임통장 목록
    └── 새 모임통장 개설 패널

└── account-detail
    ├── dashboard 탭
    ├── dues 탭
    ├── transactions 탭
    ├── members 탭
    └── settings 탭
```

## 라우팅 구조

- 전역 라우팅 상태는 `AppView = "login" | "account-list" | "account-detail"` 기준으로 관리한다.
- `login`
  - 앱 초기 진입 상태
  - 로그아웃/회원 탈퇴 이후 복귀 상태
- `account-list`
  - 로그인 또는 회원가입 성공 시 진입
  - 상세 화면에서 뒤로가기 또는 삭제 완료 후 복귀
- `account-detail`
  - 모임통장 목록에서 특정 계좌를 선택했을 때 진입
  - 내부 탭 전환은 별도 라우트가 아니라 로컬 상태로 처리

## 핵심 유저 플로우

### 1. 가입

1. 사용자는 `login` 화면에서 회원가입 모드로 전환한다.
2. 이름, 이메일, 비밀번호를 입력한다.
3. 중복 이메일이 아니면 사용자 생성 후 자동 로그인된다.
4. 앱은 `account-list`로 이동한다.

### 2. 로그인

1. 사용자는 이메일과 비밀번호를 입력한다.
2. 인증 성공 시 `account-list`로 이동한다.
3. 인증 실패 시 에러 메시지를 같은 화면에 표시한다.

### 3. 모임통장 생성

1. 사용자는 `account-list`에서 새 모임통장 개설 패널을 연다.
2. 모임명, 은행명, 계좌번호, 월 회비, 납부일을 입력한다.
3. 생성 성공 시 현재 로그인 사용자가 총무인 기본 계좌가 생성된다.
4. 생성 패널은 닫히고 목록에 새 계좌가 추가된다.

### 4. 모임통장 조회

1. 사용자는 `account-list`에서 계좌 카드를 선택한다.
2. 앱은 `account-detail`로 이동한다.
3. 기본 진입 탭은 `dashboard`다.
4. 하단 탭 또는 내부 CTA를 통해 `dues`, `transactions`, `members`, `settings`로 이동한다.

### 5. 회비 관리

1. 사용자는 `account-detail > dues`에서 월을 선택한다.
2. 멤버별 회비 상태를 확인한다.
3. 총무는 납부 상태를 `paid/unpaid`로 토글한다.
4. 화면 상단 요약과 진행률이 즉시 갱신된다.

### 6. 거래 확인

1. 사용자는 `account-detail > transactions`에서 입금/출금/전체 필터를 전환한다.
2. 거래는 날짜 그룹 기준으로 확인한다.
3. 대시보드 최근 거래 CTA에서 동일 탭으로 이동 가능하다.

### 7. 멤버 확인

1. 사용자는 `account-detail > members`에서 모임 멤버 목록을 본다.
2. 각 멤버의 역할, 연락처, 가입일, 회비 납부율을 확인한다.

### 8. 자동이체/1회성 회비/삭제

1. 사용자는 `account-detail > settings`에서 자동이체 설정을 수정한다.
2. 1회성 회비를 생성하고 멤버별 납부 상태를 토글한다.
3. 모임통장 삭제 시 계좌가 목록에서 제거되고 `account-list`로 복귀한다.

### 9. 로그아웃/회원 탈퇴

1. 사용자는 `account-list` 상단 헤더에서 로그아웃 또는 탈퇴를 선택한다.
2. 두 액션 모두 `login` 화면으로 복귀한다.
3. 탈퇴는 현재 사용자 계정 삭제까지 포함한다.

## 라우팅/탭 처리 원칙

- 화면 전환은 `AppProvider`의 전역 상태가 담당한다.
- 상세 탭 전환은 `AccountDetailScreen` 로컬 상태가 담당한다.
- 현재 구조에서는 URL 기반 딥링크보다 앱 내부 데모 플로우를 우선한다.

## 후속 구현 연결

- `GET-12`: 현재 상태 기반 라우팅을 실제 네비게이션 스택으로 확장
- `GET-22`: 화면별 데이터 필드와 액션 요구사항 구체화
- `GET-15`: 각 화면 플로우를 API 요청/응답 시퀀스로 치환
